# Desktop MCP Server Management UI — Implementation Plan

## Current State

### What Already Exists (dirty WIP in dialog-select-mcp.tsx)
- Server listing with health indicators (green/red/yellow dots via McpHealthIndicator)
- Connect/disconnect toggle switches calling sdk.client.mcp.connect/disconnect
- Add remote server form (name + URL, creates McpRemoteConfig { type: "remote", url, enabled: true })
- Auth trigger (clicking needs_auth calls sdk.client.mcp.auth.authenticate)
- Search/filter via List component
- Dropdown menu with "Delete" item (but only disconnects — does NOT remove/delete)

### What's Missing for Full CRUD Parity
1. **Proper persistence** — sdk.client.mcp.add() is ephemeral (in-memory only, lost on restart)
2. **Local server support** — form hardcodes type:"remote" with no local/command support
3. **Edit capability** — no edit form or dialog-edit-mcp component
4. **Real delete** — "Delete" only calls sdk.client.mcp.disconnect, doesn't remove config
5. **Environment variables** — no UI for env vars (local) or headers (remote)
6. **OAuth config** — no OAuth form fields (clientId, clientSecret, scope, callbackPort)
7. **Timeout config** — no timeout field per server
8. **Confirmation dialogs** — no "Are you sure?" before delete/disconnect
9. **Startup injection** — servers added via UI aren't re-created on app restart

## Key Design Decision: Persistence Strategy

**Option A — Add remove()/update() to MCP.Service + write config file:** Requires changes to Effect service layer (982 lines), HTTP API handlers, SDK client types, and SDK generated artifacts. Broad surface area touching the core opencode package.

**Option B — Store MCP configs in electron-store + inject on startup:** Follows the proven Phase 1 custom agents pattern. Narrow IPC surface, no changes to the opencode package core. The startup injection uses sdk.client.mcp.add() after the sidecar is ready.

**Decision: Option B.** It's the surgical, low-risk approach that mirrors the existing custom agents pattern exactly. The dual-source concern (config file + electron-store) is acceptable — electron-store is the desktop's persistence layer, config file is the CLI/server's persistence layer.

**Validation rules for McpServerEntry:**
```typescript
type McpServerEntry = {
  name: string
  config: {
    type: "local" | "remote"
    command?: string[]        // required if type=local
    url?: string              // required if type=remote
    environment?: Record<string, string>
    headers?: Record<string, string>
    oauth?: {
      clientId?: string
      clientSecret?: string
      scope?: string
      callbackPort?: number
      redirectUri?: string
    } | false
    timeout?: number
    enabled?: boolean
  }
}
```

**Risks:**

- Must validate entry shape to prevent corrupting the store

**Validation:**
- After adding IPC handlers, run `bun typecheck` in packages/desktop

### Step 3: Edit/Create Dialog Component
**File:**
- NEW `packages/app/src/components/dialog-edit-mcp.tsx`

**What:**
Follow the `dialog-edit-agent.tsx` pattern exactly. Component signature:
```tsx
type Props = {
  entry?: McpServerEntry        // undefined = create mode
  onSave: (entry: McpServerEntry) => void
}
export function DialogEditMcp(props: Props) { ... }
```

**Form fields (in order):**
1. **Name** — TextField (disabled in edit mode since MCP.Server is keyed by name)
2. **Type selector** — Select or Toggle between "Local (command)" and "Remote (URL)"
3. **Local fields** (shown when type=local):
   - Command — TextField (multi-valued, e.g. "npx -y @modelcontextprotocol/server-foo")
   - Environment variables — kv-pair rows (key + value TextFields, add/remove buttons)
4. **Remote fields** (shown when type=remote):
   - URL — TextField
   - Headers — kv-pair rows
   - OAuth — collapsible section with fields: clientId, clientSecret, scope, callbackPort, redirectUri
5. **Timeout** — TextField (number, ms, optional)
6. **Enabled** — checkbox or Switch (default true)

**After:**
```tsx
// dialog-edit-mcp.tsx structure
export function DialogEditMcp(props: Props) {
  const dialog = useDialog()
  const language = useLanguage()
  
  // Form state
  const [name, setName] = createSignal(props.entry?.name ?? "")
  const [type, setType] = createSignal<"local" | "remote">(props.entry?.config.type ?? "remote")
  const [command, setCommand] = createSignal(props.entry?.config.command?.join(" ") ?? "")
  const [url, setUrl] = createSignal(props.entry?.config.type === "remote" ? props.entry.config.url ?? "" : "")
  // ... more fields
  
  const handleSave = () => {
    props.onSave({
      name: name(),
      config: type() === "local" ? {
        type: "local",
        command: command().split(/\s+/).filter(Boolean),
        environment: envVars(),
        timeout: timeout() ? Number(timeout()) : undefined,
      } : {
        type: "remote",
        url: url(),
        headers: headers(),
        oauth: oauthConfig(),
        timeout: timeout() ? Number(timeout()) : undefined,
      },
    })
    dialog.close()
  }
  
  return (
    <Dialog title={props.entry ? "Edit MCP Server" : "Add MCP Server"}>
      {/* form fields */}
    </Dialog>
  )
}
```

**Risks:**
- The SDK client types (McpRemoteConfig) are generated from OpenAPI — if they diverge from ConfigMCP.Info's Effect schema, there could be type mismatches
- Command string splitting (join on " " for display, split on whitespace for storage) may break for commands with spaces in arguments

**Validations:**
- Name is required, must be non-empty
- Local: command is required, must be non-empty after split
- Remote: URL is required, must be valid URL
- Timeout: optional, must be positive number if provided

```tsx
// New imports
import { DialogEditMcp } from "./dialog-edit-mcp"
import { ConfirmDialog } from "@opencode-ai/ui/confirm-dialog"
import type { McpServerEntry, McpLocalConfig } from "@/types/mcp"

// New state
const [editingEntry, setEditingEntry] = createSignal<McpServerEntry | undefined>()
const [confirmDelete, setConfirmDelete] = createSignal<string | undefined>()

// On mount: sync electron-store MCP entries with MCP service
createEffect(() => {
  // On sync ready, load electron-store entries and add missing ones
  if (!sync.isReady || !sync.data.mcp) return
  void syncMcpWithStore()  // internal: loads from electron-store, adds missing
})

// Delete handler (replaces current handleRemove)
const handleRemove = async (name: string) => {
  setConfirmDelete(name)
}
const confirmRemove = async () => {
  const name = confirmDelete()
  if (!name) return
  // Disconnect from MCP
  await sdk.client.mcp.disconnect({ name }).catch(() => {})
  // Remove auth tokens
  await sdk.client.mcp.auth.remove({ name }).catch(() => {})
  // Remove from electron-store
  const servers = await window.electronAPI.getMcpServers()
  await window.electronAPI.setMcpServers(servers.filter(s => s.name !== name))
  // Refetch
  await queryClient.refetchQueries(queryOptions.mcp(pathKey(sync.directory)))
  setConfirmDelete(undefined)
}
```

**Risks:**
- dialog-select-mcp.tsx is already dirty (261 additions, 50 deletions) — must make additive changes only, no rewrites
- Must not break existing health indicators, toggle, auth trigger, or list rendering
- Startup sync could race against the initial server sync query

**Validation:**
- The existing toggle/connect/disconnect must still work identically
- Adding a server via DialogEditMcp must persist after refresh
- Delete must actually remove the server entry (not just disconnect)
- `bun typecheck` and `bun test` in packages/app

## Risks & Mitigations

| Risk | Severity | Mitigation |
|------|----------|------------|
| dialog-select-mcp.tsx is already dirty — conflicts with existing uncommitted WIP | High | Make strictly additive changes; never rewrite the file; read current state before editing |
| Startup sync racing with initial server sync query | Medium | Run injection after sync.data.mcp is populated (after first successful sync); use a deferred effect |
| electron-store dual-source drift (user edits config file manually) | Medium | Electron-store takes precedence for desktop-managed servers; display a source indicator |
| OAuth secrets in electron-store (plaintext) | Medium | Same as existing patterns — electron-store is local-only; no encryption requirement yet |
| Type mismatch between SDK generated types and Effect Schema types | Low | Import from SDK generated types (types.gen.ts) which is the single source of truth |
| DialogEditMcp diverging from dialog-edit-agent.tsx pattern | Low | Follow the pattern exactly — Props with optional entry, onSave callback, dialog.close() |
