# App Runtime Lifecycle v0 Specification

This document defines the runtime and execution lifecycle for Tribunus. It specifies how workspace directories, sessions, background workers, and tool calls coordinate. It establishes a strict boundary between user-facing visibility and system-level operations.

---

## 1. Directory and Session Scoping Invariant

Every session and execution context is bound to a workspace directory/project scope. 

* **Directory Scope Isolation**: All session requests must include the workspace/directory header (e.g., `x-opencode-directory`).
* **Session Invariant**: A session created under a given directory scope must be immediately and exclusively readable under that same directory scope.
* **Testing Protocol**: Verified by the `Server Runtime Test Cell v0` suite, asserting that a `POST /session` returns a session ID which is immediately fetchable via `GET /session/{id}` using matching directory headers. This guarantees cell-level isolation of active state.

---

## 2. Agent and Tool Execution Plane

The execution model divides runtime activity into three distinct layers:
1. **User-Facing Agent Runs**: High-level, sparse, and stable state representations representing user intent.
2. **Background Workers**: The operational machines executing tasks (indexing, caching, compiling).
3. **Tool Call Queue**: Coordinated execution units carrying out specific actions.

```
+------------------------------------------------------------+
|                  USER-FACING AGENT RUN                     |
|  (Durable in PGlite, sparse states: planning, executing)   |
+-----------------------------+------------------------------+
                              |
                     Projects & Coordinates
                              |
                              v
+-----------------------------+------------------------------+
|                    BACKGROUND WORKER                       |
|  (Ephemeral in Valkey: heartbeats, leases, claiming state) |
+-----------------------------+------------------------------+
                              |
                       Executes Tasks
                              |
                              v
+-----------------------------+------------------------------+
|                     TOOL CALL QUEUE                        |
|  (Durable intent in PGlite, coordinated queue in Valkey)   |
+------------------------------------------------------------+
```

### State Machine Vocabularies

#### I. User-Facing Agent Run State Machine
Represents the sparse, high-level status of the mission or session turn visible to the user:
* `created`: Session/mission has been initialized.
* `planning`: The agent is gathering context and proposing an execution strategy.
* `waiting_for_permission`: Execution is paused, waiting for explicit user approval (e.g., terminal command authorization).
* `executing`: The agent is actively executing tasks or running tools.
* `verifying`: The agent is validating the outputs of the execution (e.g., running test suites).
* `completed`: The mission succeeded, and final results/evidence are ready.
* `failed`: The mission terminated due to an unrecoverable failure.
* `cancelled`: The run was aborted by user intervention.
* `degraded`: The run is proceeding but under degraded constraints (e.g., missing non-critical dependencies or rate-limited).

#### II. Background Worker State Machine
Represents the operational status of internal, Valkey-coordinated processes:
* `registered`: Worker has started and announced its availability.
* `idle`: Worker is waiting for a stream event to claim.
* `claiming`: Worker is attempting to lease a tool call/task from the Valkey stream.
* `leased`: Worker successfully acquired the TTL lease for a task.
* `running`: Worker is executing the claimed task.
* `writing_receipt`: Worker has finished execution and is committing the result to PGlite.
* `acknowledging`: Worker is acknowledging the stream entry (`XACK`) in Valkey.
* `retry_scheduled`: Task failed but is scheduled for retry in the due-time wheel.
* `stale`: Worker failed to update its heartbeat within the lease/TTL window.
* `reclaimed`: A stale worker's lease was revoked and reassigned.
* `failed`: Worker encountered a fatal crash or internal error.
* `stopped`: Worker has been shut down gracefully.

#### III. Tool Call State Machine
Tracks individual tool execution, balancing PGlite intent durability with Valkey coordination:
* `proposed`: Tool call generated by model but not yet committed.
* `queued`: Tool call intent written to PGlite and pushed to the Valkey stream.
* `blocked_by_policy`: Tool call requires authorization or violates a policy gate.
* `leased`: Tool call claimed by a worker with an active TTL lease.
* `running`: The tool's execution is active.
* `succeeded`: Execution completed successfully.
* `failed_retryable`: Tool execution failed, eligible for retry under the scheduling wheel.
* `failed_terminal`: Tool execution failed and cannot be retried.
* `cancelled`: Execution aborted before completion.
* `receipt_recorded`: Durable result and provenance successfully written to PGlite.
* `acknowledged`: The Valkey stream entry has been acknowledged (`XACK`) by the worker.

## 3. User-Visible vs. Internal Execution (Visibility Rules)

To prevent the cockpit UI from becoming cluttered with internal execution noise, a strict boundary governs worker visibility:

1. **Hidden Healthy Internals**: By default, background workers (e.g., file watchers, indexing tasks, cache warmers, or compaction loops) remain completely invisible in the main cockpit workflow.
2. **Promotion to User Visibility**: An internal worker or tool-call state is promoted to user-facing or diagnostic UI panels *only* under the following conditions:
   * **Requires Interaction**: The execution transitions to `waiting_for_permission` or `blocked_by_policy`.
   * **Terminal Failure**: A worker or tool call encounters a `failed_terminal` state that directly impacts the user's active mission.
   * **Stale / Blocked Progress**: A worker lease becomes `stale`, or a task gets stuck, delaying the user-visible goal.
   * **Retry Storms**: Multiple consecutive retries (`retry_scheduled`) indicate degradation or rate limits.
   * **Direct Causal Relation**: The worker is directly spawned to fulfill the user's active session turn (e.g., a test runner launched during the `verifying` phase).
3. **Diagnostic Rail**: Healthy background workers are inspectable only through a separate, dedicated "System diagnostics" panel, keeping the primary session cockpit focused.

## 4. Tool-Call Queue Contract

The execution of tool calls bridges PGlite (durable truth) and Valkey (live coordination):

1. **Durable Intent**: The intent to execute a tool, including its arguments and metadata, is written to PGlite before any work is scheduled.
2. **Valkey Stream Coordination**: The tool call is enqueued as a stream entry (e.g., `tool.calls:<cellID>`) and coordinated via Valkey consumer groups.
3. **No Valkey Ack Before Durable Receipt**:
   * A worker claims the Valkey stream entry and acquires a TTL-based work lease.
   * The worker executes the tool.
   * The worker **MUST** write the tool's execution result, receipt, and provenance/artifacts to PGlite **before** executing `XACK` on the stream entry.
4. **Crash Recovery & Idempotency**:
   * If a worker crashes *before* the PGlite write, Valkey's pending list (`XPENDING`/`XCLAIM`) will flag the lease expiration, allowing another worker to reclaim and retry.
   * If a worker crashes *after* the PGlite write but *before* `XACK`, the reclaiming worker must check PGlite for a durable receipt. If found, it bypasses execution and immediately acknowledges the stream entry (`XACK`), preventing duplicate execution.

## 5. Cache Authority Taxonomy

Valkey must never become an authoritative database. The storage classes are strictly divided:

| Cache Class | Storage Engine | Examples | Recovery Behavior |
|---|---|---|---|
| **Ephemeral Cache** | Valkey Only | Model capability lists, watcher debounce states, active heartbeats, in-flight locks, provider rate-limit buckets. | Safe to lose. Re-fetched or re-created lazily on demand. |
| **Reconstructable Projection** | Valkey (Rebuilt from PGlite) | Tool call queue, due-time retry wheel, active mission indices, session run-states. | Re-projected dynamically by scanning PGlite on startup or crash recovery. |
| **Durable Cache/Receipt** | PGlite | Command execution results, file diffs, approval logs, tool call receipts, auditable model logs, provenance events. | Authoritative and persistent. Survives Valkey wipes. |

## 6. Cockpit Projection Rules

The user cockpit projects a simplified, unified state summarized from multiple fine-grained internal events:

* **Collapsing Execution State**: If one or more background workers/tool calls are in `running` or `leased` states, the user-facing status projects as `executing`.
* **Escalating Authorization**: If any active tool call transitions to `blocked_by_policy`, the user-facing state immediately projects as `waiting_for_permission`.
* **Filtered Failure**: A background worker going to `failed` does *not* transition the user-facing state to `failed` unless it causes the active tool-call queue to enter `failed_terminal` or halts progress on the user's primary mission.
* **Degraded Operations**: If a background worker experiences a `retry_scheduled` loop or rate-limiting, the user-facing state is projected as `degraded` with a non-blocking indicator.

---

## 7. Cell Federation Note

In alignment with **ADR 014 (Cell Federation)**:
* Background worker states, local heartbeats, and Valkey queue internals are strictly local to the Cell. They **never** federate across Cell boundaries.
* Federation exchanges are restricted to signed, content-light envelopes and durable receipts recorded in PGlite.
* External Cells interact via contract validation of receipts, ensuring that internal coordination mechanics (like locks or queues) remain isolated within each local execution plane.
